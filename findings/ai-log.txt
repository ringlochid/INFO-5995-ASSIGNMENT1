Evidence Collected
------------------
- `decompiled/sources/com/example/mastg_test0016/MainActivity.java`
  - Stores credentials in `credentials.txt`.
  - Contains `randomNumberGenerator()`, a non-security `Random.nextInt(100)` call that is not referenced elsewhere.
- `decompiled/sources/com/example/mastg_test0016/Login.java`
  - Reads `credentials.txt`, validates credentials, and stores `sessionToken` in `SharedPreferences`.
  - Generates the token with `java.util.Random` in `generateSessionToken()`.
- `decompiled/sources/com/example/mastg_test0016/Profile.java`
  - Clears `sessionToken` on logout.
  - Does not validate `sessionToken` before rendering the screen.
- `decompiled/resources/res/values/strings.xml`
  - The app description says the test case focuses on insecure login session handling through `Java.Util.Random`.

Rubric-Driven Mock Q&A Summary
------------------------------
Question: Which random-looking values did you identify, and which one matters?
Answer: `MainActivity.randomNumberGenerator()` is a dead-end UI/demo value. `Login.generateSessionToken()` writes a 16-character value into `SessionPrefs/sessionToken`, so that is the only security-relevant random value in the APK.

Question: Why did you change the original attack claim?
Answer: The original draft said token prediction allowed direct profile access. Re-reading `Profile.java` showed that the token is never checked on entry, so that claim was stronger than the evidence. The revised report keeps the weak-randomness finding but narrows the demonstrated impact to predictable client-side session material.

Question: What improvement most increased confidence in the submission?
Answer: Verifying every claim against the decompiled code and running a reproducible POC rather than relying on a generic explanation of `java.util.Random`.
