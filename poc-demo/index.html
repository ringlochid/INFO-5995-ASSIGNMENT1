🔓 Attacker's View – Seed Recovery Attack

Exploiting java.util.Random predictability

⚠️

Vulnerability

Login.generateSessionToken() uses new Random() which is seeded from System.nanoTime(). Because java.util.Random is a 48-bit Linear Congruential Generator, observing a token lets an attacker brute-force the seed within a small time window.

private String generateSessionToken() {
    Random random = new Random(); // ← WEAK
    StringBuilder sb = new StringBuilder(16);
    for (int i = 0; i < 16; i++) {
        sb.append(CHARSET.charAt(
            random.nextInt(62)));
    }
    return sb.toString();
}

How This Exploit Works

1
Token Generation

When a user logs in, Login.createSession() generates a 16-char token using java.util.Random seeded by the current time.

2
Attacker Observes Token

An attacker intercepts the session token (e.g. via shared preferences, network sniffing, or log exposure).

3
Seed Brute-Force

The attacker tries every plausible seed (timestamp ± window). Each candidate seed reproduces the same Random sequence. When the generated token matches, the seed is recovered.

4
Token Prediction

With the recovered seed, the attacker can predict current and future tokens, enabling session hijacking.
